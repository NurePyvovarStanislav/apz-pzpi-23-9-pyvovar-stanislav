3 ОПИС ВИКОНАНОЇ РОБОТИ
	3.1 Загальні відомості про шаблони проєктування
	Шаблони (патерни) проєктування є типовими рішеннями поширених проблем проєктування програмного забезпечення. Вони дозволяють підвищити якість коду, забезпечити повторне використання та спростити підтримку систем.
     Відповідно до класифікації GoF, патерни поділяються на: породжувальні (визначають механізми створення об’єктів), структурні (описують способи організації класів та об’єктів), поведінкові (визначають алгоритми взаємодії між об’єктам).
     Поведінкові шаблони, до яких належить шаблон Command, зосереджені на організації взаємодії між об’єктами та розподілі відповідальностей між ними. Вони дозволяють гнучко змінювати поведінку системи без модифікації її основної структури.
     Загалом використання шаблонів проєктування є важливим інструментом у сучасній розробці програмного забезпечення, оскільки вони базуються на перевірених практиках та сприяють створенню більш надійних і зрозумілих програмних систем.
     3.2 Опис обраного патерна
	Назва патерна: Command
     Тип шаблона (патерна): поведінковий
     Шаблон проєктування Command призначений для інкапсуляції запиту як окремого об’єкта, що дозволяє відокремити об’єкт, який ініціює виконання операції, від об’єкта, який безпосередньо її виконує. Такий підхід забезпечує гнучке керування запитами, їх параметризацію, а також можливість організації черги або журналу виконання команд.
     Основною ідеєю шаблона є представлення кожної дії у вигляді окремого об’єкта-команди, який містить інформацію про виклик та посилання на об’єкт-отримувач, що виконує відповідну операцію. Це дозволяє уніфікувати роботу з різними запитами через єдиний інтерфейс.
     Застосування шаблона Command є доцільним у випадках, коли необхідно:
1. відокремити логіку виклику операцій від їх реалізації;
2. забезпечити можливість розширення набору команд без зміни існуючого коду;
3. реалізувати механізми скасування або повторення дій;
4. організувати чергу або відкладене виконання операцій.
   До основних переваг шаблона належать:
1. зменшення зв’язності між об’єктами системи;
2. підвищення гнучкості та розширюваності програмного забезпечення;
3. можливість реалізації додаткових функцій, таких як журналювання або відміна операцій.
   Разом з тим використання даного шаблона має певні недоліки:
1. збільшення кількості класів у системі;
2. ускладнення структури програмного коду;
3. необхідність додаткового проєктування взаємодії між компонентами.
   Таким чином, шаблон Command є ефективним інструментом для побудови гнучких систем керування запитами, особливо у випадках, коли необхідно забезпечити слабку зв’язаність між компонентами та підтримку розширення функціональності.
     3.3 Структура шаблона (патерна)
     Шаблон проєктування Command має чітко визначену структуру, яка складається з кількох взаємопов’язаних компонентів. Кожен із них виконує окрему роль у процесі інкапсуляції та виконання запиту.
     До основних елементів шаблона належать:
1. Command – інтерфейс або абстрактний клас, який оголошує метод виконання команди (зазвичай Execute()). Він визначає єдиний спосіб виклику дій незалежно від їх конкретної реалізації;
2. ConcreteCommand – конкретна реалізація інтерфейсу Command. Містить посилання на об’єкт-отримувач (Receiver) та реалізує метод виконання, делегуючи виклик відповідному об’єкту;
3. Receiver – об’єкт, який виконує фактичну роботу. Саме він містить бізнес-логіку, що викликається командою;
4. Invoker – об’єкт, який ініціює виконання команди. Він не знає деталей реалізації, а лише викликає метод виконання команди;
5. Client – створює об’єкти команд та встановлює зв’язок між командою і отримувачем.
     Взаємодія між компонентами відбувається таким чином: клієнт створює об’єкт команди та передає йому посилання на отримувача. Після цього команда передається ініціатору, який у певний момент викликає метод її виконання. Команда, у свою чергу, звертається до отримувача для виконання конкретної дії.
     Такий підхід дозволяє повністю відокремити об’єкт, який викликає дію, від об’єкта, який її реалізує. Це забезпечує гнучкість системи та можливість легко змінювати або розширювати набір команд без модифікації існуючого коду.

	3.4 Реалізація шаблона
     Реалізація шаблона Command виконана мовою програмування C#. Для демонстрації роботи шаблона розглянуто приклад керування освітленням за допомогою пульта дистанційного керування.
     У цьому прикладі об’єкт Light виконує роль отримувача, оскільки саме він містить методи фактичного виконання дій: увімкнення та вимкнення світла. Інтерфейс ICommand визначає спільний метод Execute(), який повинна реалізувати кожна конкретна команда.
     Класи TurnOnCommand та TurnOffCommand є конкретними командами. Вони інкапсулюють відповідні дії над об’єктом Light і дозволяють викликати ці дії через єдиний інтерфейс. Клас RemoteControl виконує роль ініціатора, який зберігає команду та викликає її виконання без знання деталей реалізації.
     Основні етапи реалізації шаблона:
1. створення інтерфейсу команди;
2. створення класу-отримувача;
3. реалізація конкретних команд;
4. створення ініціатора виконання команди;
5. демонстрація роботи шаблона в основному методі програми.
     Завдяки такій структурі пульт дистанційного керування не залежить від конкретної реалізації дії. Він лише викликає метод Execute(), а конкретна команда самостійно визначає, яку операцію потрібно виконати. Це дозволяє додавати нові команди без зміни коду класу-ініціатора.
