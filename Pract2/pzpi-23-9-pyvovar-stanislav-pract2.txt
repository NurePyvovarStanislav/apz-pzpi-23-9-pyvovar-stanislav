2 ЗАВДАННЯ
	Метою практичного заняття є дослідження архітектури складної програмної системи, вивчення її структурної організації, принципів взаємодії компонентів, механізмів масштабування та забезпечення відмовостійкості в розподілених середовищах. Під час виконання роботи було обрано платформу Kubernetes як одну з найпоширеніших систем оркестрації контейнерів у сучасній хмарній інфраструктурі.
     У межах роботи необхідно було розглянути архітектурний стиль Kubernetes, проаналізувати основні компоненти системи, зокрема API Server, etcd, Scheduler, Controller Manager та Worker Nodes, дослідити механізми взаємодії між компонентами системи, побудувати UML-діаграми архітектури та розглянути приклади конфігураційних файлів і команд, що демонструють принципи розгортання та керування контейнеризованими застосунками.











3 ОПИС ВИКОНАНОЇ РОБОТИ
	3.1 Загальні відомості про систему
	Kubernetes – це відкрита програмна платформа для автоматизації розгортання, масштабування та керування контейнеризованими застосунками. Система була розроблена компанією Google на основі внутрішнього досвіду побудови великих розподілених систем та у подальшому передана для розвитку організації Cloud Native Computing Foundation.
     Основним призначенням Kubernetes є оркестрація контейнерів, тобто автоматизоване керування контейнеризованими застосунками в межах кластерного середовища. Платформа дозволяє автоматично розподіляти навантаження між вузлами, контролювати стан контейнерів, виконувати масштабування сервісів та забезпечувати відмовостійкість системи.
     Kubernetes широко використовується в сучасних хмарних та серверних інфраструктурах для побудови мікросервісних застосунків, вебплатформ, систем обробки даних та інших високонавантажених програмних рішень. Основними користувачами системи є DevOps-інженери, розробники програмного забезпечення, адміністратори інфраструктури та компанії, які використовують контейнеризацію у процесі розробки та розгортання програмного забезпечення.
     Архітектура Kubernetes побудована за принципом розподіленої системи та складається з набору взаємопов’язаних компонентів, які виконують окремі функції керування кластером. До основних можливостей платформи належать автоматичне масштабування застосунків, балансування навантаження, самовідновлення контейнерів, моніторинг стану системи та централізоване керування ресурсами кластера.
     Завдяки високій гнучкості та підтримці контейнерних технологій Kubernetes став одним із ключових стандартів у сфері хмарних обчислень та сучасної програмної інженерії.
     3.2 Архітектурний стиль системи
	Архітектура Kubernetes побудована на основі кількох архітектурних підходів, основними з яких є розподілена, клієнт-серверна та мікросервісно-орієнтована архітектура.
     Kubernetes функціонує як розподілена система, оскільки її компоненти працюють на різних вузлах кластера та взаємодіють між собою через мережеві протоколи. Такий підхід дозволяє забезпечити високу масштабованість, відмовостійкість та ефективний розподіл обчислювальних ресурсів між вузлами системи.
     В основі Kubernetes також лежить клієнт-серверна архітектура. Користувач або зовнішня система взаємодіє з кластером через клієнтські інструменти, зокрема утиліту kubectl або REST API. Усі запити передаються до центрального компонента системи – API Server, який виконує роль сервера керування кластером.
     Крім цього, Kubernetes використовує мікросервісно-орієнтований підхід. Основні компоненти системи, такі як API Server, Scheduler, Controller Manager та інші, функціонують як окремі сервіси, що виконують власні задачі та взаємодіють між собою через стандартизовані інтерфейси. Така структура спрощує масштабування, оновлення та підтримку системи.
     Важливим елементом архітектури Kubernetes є концепція Control Plane – центрального рівня керування кластером. Control Plane відповідає за моніторинг стану системи, планування запуску контейнерів, підтримку бажаного стану кластера та координацію роботи інших компонентів.
     Використання зазначених архітектурних підходів дозволяє Kubernetes ефективно працювати у великих хмарних середовищах, підтримувати тисячі контейнерів та забезпечувати автоматизоване керування складними програмними системами.
     3.3 Основні компоненти системи
     Архітектура Kubernetes складається з набору взаємопов’язаних компонентів, які забезпечують керування кластером, розгортання контейнерів та підтримку стабільної роботи системи. Основні компоненти Kubernetes поділяються на компоненти рівня керування – Control Plane, та компоненти робочих вузлів – Worker Nodes.
     Центральним компонентом системи є API Server. Він виконує роль основного інтерфейсу взаємодії з кластером та обробляє всі зовнішні й внутрішні запити. Через API Server здійснюється керування ресурсами Kubernetes, отримання інформації про стан кластера та взаємодія між компонентами системи.
     Для зберігання конфігурації та поточного стану кластера використовується etcd – розподілене сховище типу key-value. У etcd зберігаються дані про вузли, контейнери, сервіси, конфігурації та інші ресурси Kubernetes. Надійність роботи etcd є критично важливою для стабільності всієї системи.
     Компонент Scheduler відповідає за вибір робочого вузла для запуску контейнерів. Планувальник аналізує доступні ресурси кластера, навантаження вузлів та встановлені обмеження, після чого визначає оптимальний вузол для розгортання нового Pod.
     Controller Manager забезпечує підтримку бажаного стану системи. Він постійно контролює стан кластера та виконує автоматичні дії у разі виявлення відхилень. Наприклад, якщо контейнер завершує роботу аварійно, Controller Manager ініціює створення нового екземпляра.
     На робочих вузлах кластера функціонує kubelet – агент Kubernetes, який відповідає за запуск та моніторинг контейнерів. kubelet отримує команди від Control Plane та забезпечує виконання контейнерів відповідно до конфігурації Pod.
     Для мережевої взаємодії між контейнерами та сервісами використовується kube-proxy. Він відповідає за маршрутизацію мережевого трафіку, балансування навантаження та підтримку мережевих правил у кластері.
     Основною одиницею розгортання в Kubernetes є Pod. Pod являє собою групу одного або кількох контейнерів, які спільно використовують мережеві ресурси та сховище даних. Саме Pod є мінімальним об’єктом, який створюється та керується Kubernetes.
     Взаємодія зазначених компонентів забезпечує автоматизоване керування контейнеризованими застосунками, підтримку стабільної роботи системи та ефективний розподіл ресурсів у межах кластера.
     3.4 Взаємодія компонентів
     Взаємодія компонентів Kubernetes побудована на основі централізованого керування через API Server та постійного узгодження поточного стану кластера з бажаним станом системи. API Server є основною точкою взаємодії з кластером і забезпечує доступ до Kubernetes API для користувачів, внутрішніх компонентів та зовнішніх інструментів.
     Процес взаємодії компонентів можна розглянути на прикладі створення нового Pod у кластері Kubernetes. Користувач надсилає manifest-файл Deployment або Pod за допомогою утиліти kubectl. Запит надходить до API Server, який виконує перевірку та валідацію конфігурації ресурсу.
     Після успішної перевірки API Server зберігає бажаний стан нового ресурсу у сховищі etcd. etcd використовується як надійне розподілене сховище типу key-value, у якому зберігаються дані про стан кластера та його ресурси.
     Scheduler отримує через API Server інформацію про Pod, який ще не має призначеного робочого вузла. Після цього він аналізує доступні Worker Nodes, враховує обчислювальні ресурси, поточне навантаження та обмеження конфігурації, після чого призначає Pod на відповідний вузол. Інформація про це оновлюється через API Server і зберігається в etcd.
     Після призначення Pod робочому вузлу kubelet на цьому вузлі отримує через API Server інформацію про призначений Pod. Далі kubelet звертається до Container Runtime, який виконує створення та запуск контейнерів Pod. Після запуску kubelet передає статус Pod назад до API Server, а поточний стан кластера оновлюється в etcd.
     Мережеву взаємодію між Pod та сервісами забезпечує kube-proxy, який підтримує мережеві правила на вузлах і реалізує доступ до Service у кластері. Controller Manager, у свою чергу, постійно відстежує стан ресурсів Kubernetes і забезпечує наближення поточного стану системи до бажаного, наприклад шляхом повторного створення об’єктів у разі їх збою.
     Обмін даними між компонентами Kubernetes здійснюється через Kubernetes API та мережеві протоколи. Такий підхід дозволяє забезпечити узгоджену роботу компонентів системи навіть у великих розподілених середовищах.
     Для наочного представлення взаємодії компонентів використовується UML-діаграма послідовності, яка демонструє процес створення та запуску Pod у кластері Kubernetes.

     Рисунок 3.1 – UML-діаграма послідовності створення Pod у Kubernetes
     На рисунку 3.1 наведено UML-діаграму послідовності створення Pod у Kubernetes. Користувач надсилає manifest-файл через kubectl до API Server. Після перевірки запиту API Server зберігає бажаний стан ресурсу в etcd. Scheduler через API Server отримує інформацію про Pod без призначеного вузла та визначає Worker Node для його запуску. kubelet на відповідному вузлі отримує інформацію про призначений Pod і запускає його контейнери через Container Runtime. Після запуску kubelet передає статус Pod назад до API Server, а поточний стан кластера оновлюється в etcd.
3.5 Масштабованість та надійність
	Kubernetes спроєктований як система для керування контейнеризованими застосунками у розподіленому середовищі, тому масштабованість і надійність є одними з ключових властивостей його архітектури. Платформа дозволяє автоматизувати розподіл навантаження, відновлення застосунків після збоїв та зміну кількості екземплярів сервісів залежно від потреб системи.
     Одним з основних механізмів масштабування в Kubernetes є реплікація Pod. Для цього використовуються об’єкти Deployment та ReplicaSet, які визначають бажану кількість екземплярів застосунку. Якщо один із Pod завершує роботу або стає недоступним, Kubernetes автоматично створює новий екземпляр для відновлення потрібного стану системи.
     Для розподілу мережевого трафіку між кількома екземплярами застосунку використовується об’єкт Service. Він надає стабільну точку доступу до групи Pod та забезпечує балансування навантаження між ними. Завдяки цьому користувач або інший сервіс не повинен знати, на якому саме вузлі працює конкретний Pod.
     Важливою властивістю Kubernetes є самовідновлення. Система постійно порівнює поточний стан кластера з бажаним станом, описаним у конфігурації. Якщо контейнер аварійно завершує роботу, Pod стає недоступним або вузол виходить з ладу, Kubernetes може автоматично перезапустити контейнер, створити новий Pod або перенести навантаження на інший доступний вузол.
     Масштабування в Kubernetes може виконуватися як вручну, так і автоматично. Наприклад, кількість реплік застосунку можна змінити вручну через конфігурацію Deployment або за допомогою команд керування. Для автоматичного масштабування використовується Horizontal Pod Autoscaler, який змінює кількість Pod залежно від навантаження, наприклад використання процесора або пам’яті.
     Надійність системи також забезпечується розподіленою природою кластера. Застосунки можуть запускатися на різних Worker Nodes, що зменшує ризик повної відмови сервісу у разі збою одного вузла. Крім того, зберігання стану кластера в etcd дозволяє Control Plane відновлювати інформацію про ресурси та підтримувати узгодженість роботи системи.
     Таким чином, Kubernetes забезпечує масштабованість і надійність за рахунок реплікації, балансування навантаження, автоматичного відновлення, контролю бажаного стану та можливості автоматичного масштабування застосунків. Саме ці механізми роблять Kubernetes придатним для використання у великих хмарних та високонавантажених системах.
3.6 Приклади програмної реалізації
	Для демонстрації архітектурних принципів Kubernetes було розглянуто приклад конфігураційного manifest-файлу у форматі YAML. У Kubernetes такі файли використовуються для опису бажаного стану ресурсів кластера. Користувач не запускає контейнер безпосередньо, а описує, який результат має бути отриманий, після чого Kubernetes самостійно виконує необхідні дії для досягнення цього стану.
     У прикладі використовується ресурс Deployment, який визначає розгортання контейнеризованого застосунку. Deployment задає кількість реплік Pod, образ контейнера, порт застосунку та правила вибору Pod за допомогою міток. Такий підхід демонструє декларативну модель керування, що є однією з ключових особливостей Kubernetes.
     Також розглянуто ресурс Service, який забезпечує стабільний доступ до Pod та виконує балансування навантаження між їхніми репліками. Це дозволяє звертатися до застосунку через єдину точку доступу незалежно від того, на яких вузлах фактично працюють окремі Pod.
     Приклад manifest-файлу Kubernetes наведено в додатку В. У ньому описано Deployment для запуску застосунку nginx із двома репліками та Service типу ClusterIP для доступу до цих реплік у межах кластера.
     Застосування такого manifest-файлу може виконуватися за допомогою команди:
1  kubectl apply -f deployment.yaml
     Після виконання цієї команди API Server приймає опис ресурсу, перевіряє його коректність та зберігає бажаний стан у etcd. Далі Scheduler призначає Pod на відповідні Worker Nodes, а kubelet на цих вузлах запускає контейнери через Container Runtime.
     Таким чином, приклад конфігураційного файлу демонструє основний принцип роботи Kubernetes: користувач описує бажаний стан системи, а компоненти платформи автоматично забезпечують його реалізацію та підтримку.
